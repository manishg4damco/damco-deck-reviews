Section 2Where D365 Falls Short
D365 Was Built for Desktops. Your Teams Aren't.
The standard D365 approval interface creates real friction for teams that need to act fast, from anywhere, on any device. Five pain points, all documented and solvable.
1
Desktop-First Design
No mobile-optimised interface. Approvers must navigate full D365 UI on a phone: small touch targets, horizontal scrolling, multi-tab navigation. Approvals get deferred or skipped.
2
No Unified Approval Dashboard
Each approver must navigate to separate D365 screens for PR and PO approvals. There is no single view of all pending actions. Items get missed.
3
Query and Hold Actions Are Buried
The Query and Hold options are several clicks deep in the D365 approval UI. Approvers take the path of least resistance; they approve or reject, even when a query is the right action.
4
Requestors Have No Visibility
Once a PR or PO is submitted, the requestor has no real-time status. They must dig into D365, or chase approvers directly, to find out where their request stands.
5
Escalation Is Invisible
If an approver is on leave or inactive, there is no automated escalation. The workflow stalls silently. No one is notified until someone raises the alarm manually.

Section 2How the D365 Workflow Engine Works
Sequential Approval Steps
Each step completes before the next begins. Approval order is configured in D365 and enforced automatically.
Conditional Routing
Route by amount threshold, department, financial dimension, or legal entity. Different PRs follow different paths without manual intervention.
Time Limits per Step
Each approval step has a configurable SLA in hours or days. When the limit is breached, D365 escalates to the next approver in hierarchy.
Completion Policies
Single approver, majority vote, all approvers, or percentage-based. Configurable per step. D365 enforces, the app surfaces.
Workflow States in D365 F&O
Draft PR created, not yet submitted. Requestor can still edit. No workflow instance active.
Submitted PR entered the workflow. D365 identifies the first approver from the configured hierarchy.
In Review Sitting with an approver. SLA timer is running. Action is pending.
On Hold Approver has paused the step (budget pending, additional sign-off needed). SLA timer paused. App label.
Queried Approver requested clarification. D365 triggers "Request Change" internally. PR returns to Draft. App label.
Approved All approval steps complete. PR is ready for PO conversion. D365 native state.
Rejected Returned with mandatory rejection reason. PR is closed. Full audit trail preserved in D365.

Section 4.3Key App Features
Mobile-First UI
Native iOS, Android, and responsive web. Built for one-thumb operation; approvals in under 10 seconds.
Unified Pending Actions
All PRs and POs requiring action in one dashboard, sorted by SLA urgency, filterable by entity and category.
One-Tap Actions
Approve, Reject, Query, Hold, and Delegate. All accessible with a single tap, with inline comment entry.
Document Viewer
View attachments (purchase orders, invoices, quotes) inline within the approval screen, without leaving the app.
Real-Time Status Tracking
Requestors see live status of their submitted PRs and POs: current approver, stage, and expected SLA.
Delegation & Out-of-Office
Approvers set delegation periods; the system automatically routes to the delegate and notifies all parties.
SLA Monitoring
Configurable SLA thresholds per document type. Auto-escalation to next approver if SLA is breached. Full audit trail.
Audit Trail
Every action (approval, rejection, query, delegation) is logged with timestamp, user, and comment for compliance.

Section 5.2Data Flow Paths
Path A: Read
OData REST API
Fetching pending items and document details
1
User opens the app, authenticated via Azure AD OAuth 2.0
2
Power Apps queries D365 for pending workflow items
GET /data/WorkflowWorkItems?$filter=Status eq 'Pending'
3
D365 OData returns PR and PO headers with financial details
GET /data/PurchaseRequisitionHeaders
4
App renders the Pending Actions dashboard, sorted by SLA urgency
5
Document attachments fetched on-demand via Document Management API
Path B: Action
Workflow Action API
Approving, rejecting, or querying a document
1
Approver taps Approve (or Reject / Query / Hold) with optional comment
2
Power Apps calls Power Automate Action Handler flow via HTTP connector
3
Action Handler POSTs to D365 WorkflowWorkItem-complete action
POST /data/WorkflowWorkItems('{id}')/Microsoft.Dynamics.DataEntities.WorkflowWorkItem-complete
4
D365 updates the workflow state, advancing to next stage or closing
5
Confirmation returned to Power Apps. Dashboard refreshes immediately.
Path C: Event
Business Events
Real-time notifications triggered by D365 state changes
1
A workflow state change occurs in D365: new item assigned, SLA threshold reached, or escalation triggered
2
D365 fires a Business Event, pushed to the configured endpoint
WorkflowWorkItemCreated / SLABreach event
3
Power Automate Event Listener flow receives the event payload
4
Flow routes notification (Teams, push, or email) to the correct recipient
5
For SLA breaches: Escalation Handler fires, reassigns document and logs audit entry

Section 5.3Security and Technology Stack
Authentication and Data Security Model
1
Azure AD Single Sign-On
Users log in with their existing LensKart credentials. No new identity setup. Azure AD is already live at LensKart.
2
OAuth 2.0 Bearer Token
Azure AD issues a per-user Bearer token. Token is passed on every API call. Power Platform refreshes tokens automatically.
3
D365 RBAC Enforcement
D365 validates the token on every call and applies row-level and column-level security. Users see only what their D365 role permits.
4
Singapore Entity Scope
OData queries are scoped to the Singapore legal entity for the POC. Cross-entity data does not bleed across. Controlled via D365 cross-company access settings.
Role-Based App Views
Requestor My Submissions: PR/PO status, approver identity, timeline
Approver Approval Inbox: pending items sorted by SLA urgency
Manager Team Overview: escalations, SLA breaches, pending by approver
