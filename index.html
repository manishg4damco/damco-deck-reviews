Vendor Brief Responses
Answers to Your Questions
Q1 — OData API vs direct SQL: which integration approach?
OData + BYOD analytics database. Not direct SQL on AXDB.

Microsoft's official guidance for D365 F&O on-premises identifies OData via Data Entities as the supported integration pattern. Direct AXDB access is unsupported, introduces locking risk on live ERP operations, and breaks on schema changes during system updates.

OData handles real-time lookups on small result sets. The BYOD analytics database handles analytical aggregations — trend analysis, multi-entity comparisons — on a performance-isolated read-only replica. The two mechanisms work together as one data layer.

Q2 — Should a data warehouse layer be introduced? How is data kept current?
Yes, a lightweight BYOD analytics database. Not a full star-schema warehouse.

A dedicated SQL Server 2022 instance on-premises receives incremental delta exports from D365 via BYOD batch jobs. After the initial full load, only changed records are pushed — no full-table refreshes.

A full data warehouse adds 3 to 4 months of complexity for capabilities BYOD already covers. It can be introduced in Phase 2 if cross-system analytics become a requirement.

Q3 — Is the agent limited to a fixed set of questions?
No. Open-ended NL queries across all four D365 modules.

The agent is a generalised NL-to-SQL engine with a semantic layer. It does not have a pre-compiled list of supported reports. Any well-formed financial or operational question can be answered in English or Arabic from the same interface.

The semantic layer is what makes this possible. It maps IKK's business vocabulary to D365's schema: what "revenue" means in IKK's chart of accounts, how legal entities join to financial dimensions, and how fiscal calendar filters apply. The semantic layer is built for IKK's specific D365 configuration during Phase 1.

Full use case coverage across all four modules is on the following slide.

Q4 — What hardware is recommended?
HPE ProLiant DL380a Gen11 with 2x NVIDIA L40S 48 GB.

The DL380a Gen11 is the purpose-built AI/GPU variant of the DL380 family, designed for sustained inference workloads. Two L40S cards provide 96 GB total VRAM — sufficient for Qwen2.5-72B at 4-bit quantisation with headroom for concurrent sessions and the full application stack.

A separate HPE DL360 Gen11 is recommended as the dedicated Analytics SQL Server, preventing resource contention during heavy query periods. Full specifications and the extended capacity configuration are on the next slide.
